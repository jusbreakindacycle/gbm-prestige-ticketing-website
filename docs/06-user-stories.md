# Deliverable 06: User Stories
## Homeland Travel and Tours – GBM Prestige Ticketing Website (v1)

---

# 1. Document Purpose

This document translates the approved use cases and requirements into user-centered statements. It expresses each planned interaction from the perspective of the person performing it, making it easier to confirm that every feature in v1 serves a real user need before development begins in VS Code.

---

# 2. User Story Overview

A user story describes what a user wants to do, why they want to do it, and what result they expect. Unlike a use case, which describes the flow of an interaction, a user story keeps the focus on the user's intent and the value they get from completing the action.

---

# 3. User Story Format

All user stories in this document follow this format:

> As a **[user type]**, I want to **[goal]**, so that **[benefit]**.

---

# 4. User Personas / User Types

### Potential Customer
A first-time visitor evaluating whether to trust and contact the business.

### Existing Customer
A returning visitor who has used the business before and wants quick reference or fast contact.

### Business Owner / Staff
The person(s) who receive inquiries generated by the website; benefits indirectly from clearer, better-informed visitor questions.

### Mobile Visitor
Any visitor (potential or existing customer) accessing the site primarily from a smartphone.

### Search Visitor
Any visitor who discovers the site through a search engine rather than a direct link or social media referral.

# 6. Acceptance Criteria per User Story

**US-01 — View Homepage**
- Homepage loads correctly at the site's root URL.
- Business name, slogan, and short introduction are visible without scrolling excessively.
- Links to Services and Contact are visible on the homepage.

**US-02 — Learn About the Business**
- About page is reachable from the main navigation.
- Content describes the business background consistent with approved documents.

**US-03 — Understand the 8-in-1 Ticketing System**
- Homepage and FAQs page both explain that "8-in-1" refers to 8 assisted services, not a software system.
- Explanation uses simple, non-technical language.

**US-04 — Browse All Services**
- Services page lists all 8 services with a short description each.
- Each service includes a clear inquiry CTA.

**US-05 — View Tour Package Information**
- Tour Packages page is reachable from navigation or the Services page.
- Content presents general categories/examples without implying fixed pricing or real-time availability.

**US-06 — Read FAQs**
- FAQs page is reachable from the main navigation.
- Includes at minimum: meaning of "8-in-1," how to inquire, and confirmation that bookings/payments aren't handled on the site.

**US-07 — Find Contact Details**
- Contact page lists phone number and Messenger, Viber, WhatsApp, and Telegram links.
- Contact page is reachable from the main navigation.

**US-08 — Use Contact Buttons**
- Contact buttons appear on Homepage, Services, and Tour Packages pages.
- Each button opens the correct app or dialer when clicked/tapped.

**US-09 — Navigate on Desktop**
- Navigation menu is visible and functional on desktop screen widths.
- All main pages (Home, About, Services, Tour Packages, FAQs, Contact) are reachable from the menu.

**US-10 — Navigate on Mobile**
- Navigation collapses into an accessible menu (e.g., toggle/hamburger) on mobile widths.
- All pages remain reachable, and layout adjusts without horizontal scrolling.

**US-11 — Find the Website Through Search Engines**
- Each page has a descriptive title and meta description.
- Content uses clear heading structure to support indexing.

**US-12 — Recover from a Broken or Invalid Page**
- A custom "page not found" message appears for invalid URLs.
- The message includes a link back to the homepage.

# 7. Priority Guide

- **High:** Essential for v1 launch; the site does not meet its core purpose without this story being satisfied.
- **Medium:** Adds meaningful value and should be included where time and effort allow, but the site can still function without it.
- **Low:** Optional polish that is not necessary for v1's core purpose.

# 9. Scope Boundaries

The following user stories are intentionally excluded from v1, consistent with the approved SRS, Functional & Non-Functional Requirements, and Use Case Analysis:
- User registration
- Login
- Booking submission
- Online payment
- Inquiry form submission
- Database-backed inquiry tracking
- Admin dashboard actions

No story in this document assumes the visitor creates an account, submits a form, completes a booking, or makes a payment on the website. Every contact-related story (US-07, US-08) ends with the visitor being handed off to an external messaging app or phone call, consistent with the site's static, inquiry-only scope.

---

# 10. Summary

This User Stories document translates the approved use cases and requirements into 12 user-centered stories across five user types: Potential Customer, Existing Customer, Business Owner/Staff, Mobile Visitor, and Search Visitor. Each story includes acceptance criteria and traces back to its supporting use case, functional requirement, and project goal. All stories remain within the static, zero-budget, inquiry-focused scope already established, with no story depending on registration, login, bookings, payments, forms, or database-backed features. This document is ready to guide page content, CTA wording, and navigation behavior in future VS Code development.
